Vim

Vim is a minimalistic text editor that is built into Unix and is often the default text editor within the terminal.

Key Features:

  1. Modal Editing:
  • Normal Mode: Navigate and manipulate text.
  • Insert Mode: Insert and edit text.
  • Visual Mode: Select and highlight text.
  1. Commands
  • Performs complex text manipulations with minimal keystrokes.
  • Examples: dd (delete line), yy (yank/copy line), p (paste).

Vim is pretty old school but knowing a couple of commands can help you out in some case (for example with merge conflicts, which we will get to when we talk about git).

Basic vim commands

Opening and exiting vim

  • vim filename: open a file in vim
  • :w: save changes
  • :q: quit vim
  • :wq: save changes and quit

Basic vim commands

Navigating in vim

  • h, j, k, l: move left, down, up, and right
  • gg: go to the beginning of the file
  • G: go to the end of the file

Editing text

  • i: enter Insert mode before the cursor
  • a: enter Insert mode after the cursor

  • Can move files back and forth from local computer to cluster using scp
  • This is not ideal!
    • Probably won’t get the code right the first (or second, or even third time) and doing scp gets cumbersome quick
    • Hard to keep track of what code is local and what code is on the cluster
    • Poor version control

This is what I did as a grad student so I know first hand that it can be super annoying

Project organization on the cluster

Solution: create an organized file structure and put it on GitHub!

You can then commit changes locally, push them to GitHub, and pull them down in the cluster

  • Easier than scp
  • More reproducible
  • Version control as a byproduct

Steps for starting a cluster project

Once you have code in a well structured directory that works the way you expect it to work, do the following:

  • Initialize local repo with desired structure
  • Create .gitignore file
    • Should contain types of files you don’t want to push back and forth between local computer and GitHub
  • Create empty remote repo of same name on GitHub
  • Link remote repo to local repo and push local contents
  • Log into cluster and clone the remote repo

For local repo test your code very thoroughly to make sure it’s doing what you expect before you start this process Also, I really only push in one direction here

Making changes to your code

Let’s say we want to increase the number of simulations or change the study parameters.

  • Make the changes to your repo locally
  • Commit and push to GitHub
  • Pull them down on the cluster

Do this.

Useful commands

  • Check current running jobs for a user: squeue -u <your_user_name>
    • For example, my user name is jwrobe8
  • Check current running jobs for a partition: squeue -p <partition_name>
  • Kill a job: scancel <JOBID>
  • Kill all jobs: scancel -u <your_user_name>

Useful commands

  • Copy from remote to local: scp -r <your_user_name>@clogin01.sph.emory.edu:/<path_to_file> .
  • Copy from local to remote: scp <file_name> <your_user_name>@clogin01.sph.emory.edu:/<path_to_file> .
    • -r flag here is not necessary but it is useful; it means “recursive” and can be used to copy whole directories

Allocating memory

If your job fails and it’s not due to a bug in the code, it could be due to not having enough memory. You can fix this by increasing the amount of memory allocated to each job. Put the following in the header of your .sh file:

# SBATCH --mem=40G for specifying

Note: Be careful with this. Sometimes it is tough to know how much memory to allocate, so start by specifying nothing (I think the default is about 10G but don’t quote me on that).
